## Overclaim 유형 정의

| 유형 | 설명 | 위험도 |
|------|------|--------|
| **A. 자동평가 위장** | 외부 도구가 필요한데 자동 평가처럼 보이게 함 | 🔴 높음 |

| **D. 문서 의미 단순화** | 문서의 절차/맥락을 생략하여 의미 변질 | 🟡 낮음 |

## 점검 결과

### 🔴 A. 자동평가 위장 — 없음 ✅

| 점검 항목 | 결과 |
|----------|------|
| 사전 계산 업로드형을 "자동 평가"로 표시했는가 | 아니오 — "사전 계산 업로드" 배지 사용 |
| LLM-as-a-Judge를 정적 사이트에서 실행하는 것처럼 보였는가 | 아니오 — "외부 LLM Judge가 사전 평가한 결과를 업로드" 안내 |
| Red Team ASR을 자동 테스트처럼 보였는가 | 아니오 — 수동 공격 결과 입력형 |
| 벤치마크를 실시간 실행하는 것처럼 보였는가 | 아니오 — 사전 실행 결과 입력형 |

**결론**: A유형 overclaim 없음.

### 🟠 C. 지표/기법 왜곡 — **3건 발견**

#### C-1. LLM-M-02 정확성: ROUGE/BERTScore → 문서는 ACC(MMLU) 🔴

| | 문서 | 구현 |
|---|------|------|
| 평가 지표 | ACC (정답률) | "0~1 스코어" |
| 검증 기법 | 벤치마크 테스트 (MMLU) | "Reference comparison (ROUGE, BERTScore)" |
| 벤치마크 | MMLU | 미언급 |

**위험**: 사용자가 ROUGE/BERTScore로 정확성을 평가해야 한다고 오해할 수 있음.
문서는 "정답이 존재하는 경우에 한정해 정답과의 일치 여부"를 ACC로 측정하라고 명시.
ROUGE/BERTScore는 문서 어디에도 없음.

**심각도**: 🔴 높음 — 평가 방법 자체가 다름
**수정 방안**: GenerativePerformance.tsx의 정확성 탭 설명을 "벤치마크 테스트 (MMLU) 기반 ACC (정답률)"로 수정

#### C-2. LLM-M-03 유창성: Perplexity 병기 🟠

| | 문서 | 구현 |
|---|------|------|
| 평가 지표 | Likert 척도 | "1~5 Likert" ✅ |
| 검증 기법 | LLM-as-a-Judge (G-Eval) | "Perplexity or LLM-as-Judge" ⚠️ |

**위험**: 이전 repo(NB05)에서 사용한 Perplexity가 잔재로 남아 있음. 문서는 Perplexity를 언급하지 않음.
사용자가 Perplexity를 공식 지표로 오인할 수 있음.

**심각도**: 🟠 중간 — Likert가 주 지표이긴 하나 Perplexity가 병기됨
**수정 방안**: "Perplexity" 관련 텍스트 제거. G-Eval 프롬프트 예시(문서 p.? 예시) 추가.

#### C-3. LLM-M-01 사실성: "External fact-checking API" 표기 🟠

| | 문서 | 구현 |
|---|------|------|
| 검증 기법 | 벤치마크 테스트 (TruthfulQA) | "External fact-checking API or LLM-as-Judge" |
| 평가 지표 | Truthfulness % | "0~1 score" |

**위험**: 문서는 TruthfulQA라는 특정 벤치마크를 명시. "fact-checking API"는 문서에 없는 용어.
사용자가 임의의 fact-checking API를 사용해도 된다고 오인할 수 있음.

**심각도**: 🟠 중간
**수정 방안**: "벤치마크 테스트 (TruthfulQA) 기반 Truthfulness %"로 수정

### 🟡 D. 문서 의미 단순화 — **4건 발견**

#### D-1. ML-M-03 설명가능성: LIME 누락 및 xAI 선정 안내 없음

**문서**: "모델 알고리즘을 고려하여 적합한 xAI 기술을 적용 (예: 트리 기반 → TreeSHAP, 딥러닝 → DeepSHAP)"
**구현**: SHAP만 다루고 LIME은 언급하지 않음. xAI 기술 선정 가이드 없음.

**심각도**: 🟡 낮음 — 기능적으로는 SHAP 결과 업로드로 충분하나, 문서 충실도 부족.
**수정 방안**: 설명 텍스트에 LIME 안내 추가. "모델 유형별 xAI 선택 가이드" 섹션 추가.

#### D-2. LLM-M-08 간결성: "Length vs information density" 설명

**문서**: "장황하지 않고 핵심적으로 요약되는지 여부" + "LLM-as-a-Judge (G-Eval)" + "Likert 척도"
**구현**: "Length vs information density (0~1 score)"

**심각도**: 🟡 낮음 — 본질적 의미는 유사하나 표현이 문서와 상이.
**수정 방안**: "LLM-as-a-Judge (G-Eval) Likert 1~5 척도"로 통일.

#### D-3. LLM-S-05 안전장치 우회: 공격 전략 카탈로그 미제공

**문서**: [표 2.3.13]에서 7가지 공격 전략 상세 설명 (완곡한 표현, 아첨, 미끄러짐, 모델조작, 질문쪼개기, 역할극, 비전통적 형식)
**구현**: 공격 케이스 입력 테이블만 제공. 공격 전략 유형 안내 없음.

**심각도**: 🟡 낮음 (기능에는 영향 없으나 실습 교육 가치 감소)
**수정 방안**: 공격 전략 카탈로그를 정적 텍스트/카드로 표시.

#### D-4. LLM-S-06 환각: FEVER 데이터셋 구조 미안내

**문서**: FEVER Score = SUPPORTED/REFUTED/NOT ENOUGH INFO 분류 정확도
**구현**: 단순 스코어 입력 (0~1)

**심각도**: 🟡 낮음
**수정 방안**: FEVER 데이터셋 구조(3-class classification) 안내 텍스트 추가.

---

### 🟡 E. 시각적 과장 — **1건 발견**

#### E-1. GenerativePerformance 탭 구조

**현상**: 8개 성능 지표가 탭으로 나열되어 있으나, 사실성·정확성·유창성·관련성·일관성·완전성·간결성 7개 탭은 모두 동일한 "스코어 입력 + configurable threshold" UI.
다양성(Distinct-N)만 실제 계산 기능 제공.

**위험**: 7개 탭이 "모두 실습 가능"한 것처럼 보이지만, 실제로는 값 입력 후 threshold 비교만 수행.
이것 자체가 overclaim은 아니지만, "인터랙티브 실습"이라는 기대와의 괴리 존재.

**심각도**: 🟡 낮음 — 각 탭에 "사전 계산 업로드" 배지가 표시되어 있으므로 정직하게 처리됨.
**개선 방안**: 각 탭에 "이 지표는 외부에서 사전 평가한 결과를 입력하는 형태입니다" 안내 텍스트 강화.
